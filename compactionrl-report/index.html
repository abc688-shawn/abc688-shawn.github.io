长程 agent 的难点不只是“上下文放不下”，而是 一旦历史被摘要替换，后续所有动作都被迫相信这份摘要。如果 RL 只奖励最后任务成功，却没有训练摘要本身，压缩就会成为训练管线外的一段脆弱手工逻辑。

CompactionRL 的判断是：摘要不是日志归档，而是一类会影响最终 reward 的 policy action。于是摘要 token 和执行 token 被放进同一个 PPO 训练问题里，论文再处理两个随之出现的偏差：分段数量/长度造成的 loss 权重偏差，以及 terminal reward 被错误拉近的信用分配偏差。

THESIS

我的读法：这篇论文最值得看的不是“模型学会写更长摘要”，而是它把上下文管理从 prompt engineering 移到 RL 轨迹结构内部。它真正讨论的是：当一个 rollout 被 summary 切成多段之后，PPO 的样本、token 权重和 GAE 距离应该如何重新定义。

03 · METHOD ANATOMY

CompactionRL 把一次长 rollout 改写成“执行段 + 摘要段”的训练对象

论文的设置很具体：agent 在软件工程、终端任务这类长程交互中不断产生 assistant response 和 environment observation。历史越长，越容易逼近固定 context budget。CompactionRL 不把 summary 当作外部工具，而是在 rollout collection 时由当前 policy 生成 summary，再从 summary 和最近若干轮交互恢复上下文。

  1. 原始历史按 assistant-observation 原子步组织。论文把 \(z_i=(a_i,o_i)\) 当作不可拆分单元，避免 compaction 把工具调用和工具反馈切开。
  2. 预算不足时触发 summary。当剩余 context budget 小于 \(T_{\mathrm{comp}}\) 时，policy 在当前历史后追加固定总结指令并采样 summary。
  3. 恢复上下文保留两类信息。长程信息靠 summary，最近环境状态靠尾部 \(k\) 个原子步。论文默认 \(k=2\)，必要时减少以满足预算。
  4. 训练时 summary token 与 execution token 都被优化。每条 compacted rollout 拆成若干 segment，但它们共享同一个最终任务 reward，不另造 summary-quality reward。
Design tension 压缩让 rollout 继续变长，但也让一个任务样本变成可变数量的训练 segment。论文选择 PPO 而不是 GRPO 式 group-wise normalization，正是因为固定组大小的假设在 compaction 后会变得别扭。
Paper-backed

机制来自 Section 4.1 与 Figure 2：summary 由同一 trainable policy 生成，final task reward 分配给同一 rollout 的全部 trainable segments。

Caveat

论文没有把完整总结 prompt、环境实现细节和所有失败案例展开，因此复现仍需要源码/训练管线级信息。

04 · OBJECTIVE

公式要解决两件事：别让长短 segment 改变权重，别把远处 reward 看近了

CompactionRL 的公式部分可以分成三层。第一层定义何时压缩与如何恢复；第二层把 PPO loss 写到 token 粒度；第三层用跨轨迹位置校正，让早期 segment 的 advantage 不会因为“每段独立训练”而错误地靠近终点奖励。

Context compaction trigger and resume
\[ h_t=(s,u,z_1,\ldots,z_t), \quad z_i=(a_i,o_i) \] \[ C-|h_t| < T_{\mathrm{comp}}, \quad S_t \sim \pi_\theta(\cdot \mid h_t \oplus q_{\mathrm{sum}}) \] \[ \bar{h}_t=(s)\oplus u_{\mathrm{resume}}(S_t)\oplus(z_{t-k+1},\ldots,z_t) \]
这里 \(C\) 是峰值 context budget，\(T_{\mathrm{comp}}\) 是触发阈值，\(S_t\) 是 policy 生成的 summary。论文的关键设计是：恢复上下文不是只用 summary，而是 summary 加最近 \(k\) 个 assistant-observation 原子步。
Token-level PPO objective
\[ \rho_{s,i}(\theta)= \frac{\pi_\theta(y_{s,i}\mid x_{s,i})} {\pi_{\theta_{\mathrm{old}}}(y_{s,i}\mid x_{s,i})} \] \[ \mathcal{L}_{\pi}= -\frac{1}{|\mathcal{M}|} \sum_{(s,i)\in\mathcal{M}} \min\left( \rho_{s,i}(\theta)\widehat{A}_{s,i}, \mathrm{clip}(\rho_{s,i}(\theta),1-\epsilon,1+\epsilon)\widehat{A}_{s,i} \right) \]
论文把平均单位放在 optimized assistant-token set \(\mathcal{M}\) 上，而不是 sequence 或 segment 上。我的理解是，这一项在防止“更容易触发 compaction 的任务，因为 segment 更多而在训练中更吵”。
Cross-trajectory GAE correction
\[ A^{\mathrm{loc}}_{s,i}= \sum_{\ell=0}^{n_s-i}(\gamma\lambda)^\ell\delta_{s,i+\ell}, \quad \delta_{s,i}=r_{s,i}+\gamma V_\phi(x_{s,i+1})-V_\phi(x_{s,i}) \] \[ N_{>s}=\sum_{j>s} n_j,\quad \widehat{A}_{s,i}=(\gamma\lambda)^{N_{>s}}A^{\mathrm{loc}}_{s,i} \]
如果每个 segment 独立计算 GAE，terminal reward 会像是出现在每段末尾，早期 summary 被过度奖励。乘上 \((\gamma\lambda)^{N_{>s}}\) 等于告诉训练：这个 token 到最终 outcome 之间还隔着后续 segment 的 optimized tokens。
CLAIM

summary 不是辅助标签

论文没有引入手写 summary quality reward，而是让 summary token 直接受最终任务 reward 影响。

WHY PPO

可变 segment 数破坏 group 假设

GRPO 类 group-wise 方法在 segment 数随任务变化时，会遇到重复计权和 segment-level advantage 的问题。

LIMIT

GAE 仍是近似

论文限制部分承认 cross-trajectory GAE 不是精确 full-trajectory credit assignment。

06 · BEHAVIOR

行为图说明：收益不像是单纯“多跑几步”，而是摘要训练让继续执行更稳

论文 Figure 3 观察 GLM-4.5-Air 在 80k×4 compacted evaluation 下的行为。CompactionRL 的 compaction 次数和 tool call 数不是最多，compacted-task accuracy 却最高。作者据此认为，提升不是因为它简单拉长 trace，而是 trained summaries 减少了压缩后重新探索和状态丢失。

Figure 4 则看训练动态：当 summary response 进入 RL objective，summary length 随训练变长，reasoning tokens per turn 增加，entropy 上升更慢。论文的解释是，模型学会保存更多可继续执行的状态，而不是只写概括性进度。

我的综合判断：CompactionRL 的实验最强证据来自“summary train + token-level loss + cross-trajectory GAE”三者同时出现时的 compacted performance，而不是来自单个漂亮曲线。
Synthesis

行为图与 ablation 互相补强：summary 训练让 compacted state 更有用，优化修正让这些 summary token 得到较合理的训练权重与信用距离。

Unproven

论文没有证明更长 summary 本身总是更好；它只显示在该训练和 benchmark 设置中，summary length 增长伴随更好的 compacted behavior。

07 · LIMITS AND OPEN QUESTIONS

真正的边界：它训练的是 compaction-enabled agent，不是通用 single-window 增强器

论文自己的 limitations 很直接：CompactionRL 针对启用 compaction 的执行方式训练，关闭 compaction 后 gains 不稳定。这不是小脚注，而是决定部署方式的前提。若生产环境不允许 test-time compaction，这套方法可能不会自然迁移。

哪些结论是 paper-backed？

CompactionRL 在论文设置的 compacted evaluation 下提高 SWE-bench Verified 与 Terminal-Bench 2.0 Pass@1；summary agent 的选择会影响任务成功；去掉 token-level loss 或 cross-trajectory GAE 会降低 GLM-4.5-Air-SFT 的 80k×4 结果。

哪些是我的推断？

这篇论文的长期价值可能在“重定义长程 RL 样本结构”，而不只是上下文压缩技巧。这个判断来自方法章节和 ablation 的组合，不是论文直接宣称的定理。

仍未证明什么？

跨分段 GAE 只是近似；coding-oriented benchmark 之外的任务域未覆盖；更长 summary 与更好任务成功之间没有被拆成因果机制；公共模型 baseline 因 scaffold 差异不能直接横向排位。

下一步该看什么？

读者可以接着比较 ReSum、SUPO、Context-Folding 一类 context management RL 工作，重点看 summary 是外部工具、训练动作，还是 branch-and-fold 技能；再看各自如何处理 reward attribution。

Bottom line 如果你在训练长程 coding agent，CompactionRL 提供了一个具体配方：压缩必须进 rollout collection，summary token 必须进 objective，segment 化后的 PPO 必须处理权重和时间距离。若只是想在推理时“把历史总结一下”，这篇论文反而提醒你：summary 质量可能已经是任务性能瓶颈。
Limitations

对应论文 Conclusion 的 Limitations：single-window transfer 不稳定、cross-trajectory GAE 近似、实验域主要是代码长程任务。

Open question

如果 summary 本身会变长，何时会重新吃掉 compaction 节省下来的预算？论文只给了当前设置下的积极迹象。

更大的图景

长程 agent 的上下文问题，正在从“能不能装下更多 token”转向“哪些历史状态值得被训练成可继承的记忆”。CompactionRL 的答案很工程化：让模型自己写交接摘要，并让最终任务成败回流到这些摘要 token。
