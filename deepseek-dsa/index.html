从全量注意力到 DeepSeek Sparse Attention

DSA 的核心不是让模型“少想”，而是让模型先学会找到最该看的历史，再把昂贵的主注意力花在这些位置上。

一句话结论

DeepSeek Sparse Attention，简称 DSA，本质上是在 Transformer 的自注意力里加了一个很轻的检索器：每个新 token 不再让主注意力层认真看完所有历史 token，而是先用 lightning indexer 为历史 token 打分，只挑 top-k 个最可能有用的 KV 条目，再让真正的注意力只在这些条目上做 softmax 和加权求和。

它没有换掉自回归语言模型。tokenization、embedding、RoPE、RMSNorm、残差、MoE FFN、LM head、逐 token 生成都还在；真正变化的是主注意力从“全量读历史”变成“先学会找，再精读”。
1

为什么会走到 DSA

注意力机制的演进可以看成一连串针对同一个矛盾的回应：上下文越长，模型越需要远距离信息；但越让每个 token 看完整历史，成本越不可承受。

2017
Transformer
全量自注意力让任意 token 可直接关联，但训练和 prefill 的注意力矩阵随 L² 增长。
2019
MQA
多个 query heads 共享 K/V，降低自回归 decode 的内存带宽压力，但仍可能看全部历史位置。
2020
Longformer / BigBird
固定窗口、全局 token、随机或块状稀疏连接把长文档变得可处理，但模式不够语义自适应。
2022
FlashAttention
通过 IO-aware tiling 把 exact dense attention 做快，但数学上仍是全量注意力。
2024
DeepSeek-V2/V3 的 MLA
把每个 token 的 KV 压成 latent vector，大幅降低 KV cache，但历史位置数量本身仍随上下文增长。
2025
NSA 与 DSA
NSA 验证硬件友好、可训练稀疏注意力；DeepSeek-V3.2 通过 continued training 把 DSA 接入 MLA。
2026
DeepSeek-V4 CSA/HCA
先沿序列维度压缩 KV，再在压缩条目上使用 DSA 或 dense attention，使 1M context 更实际。
2

上一代技术卡在哪里

一句话压缩：
全量注意力的问题是位置太多，每个位置都要互相看。MQA/GQA 让每个位置的 K/V 少存一点；MLA 让每个位置的 K/V 压得更小；FlashAttention 让全量计算执行更快；DSA 则让模型自己决定每个 query 应该看哪些历史位置。

旧路径已经解决的事

  • FlashAttention 降低 dense attention 的显存和 IO 开销。
  • MQA/GQA 降低 decode 阶段 KV cache 读取压力。
  • MLA 压缩每个 token 的 KV 表示。
  • 早期 sparse attention 证明长序列不必总是全量连接。

仍然留下的痛点

  • 历史位置数量仍然随上下文线性增长。
  • 固定稀疏模式可能漏掉远处但关键的 token。
  • 长工具轨迹和长推理链会让每步生成越来越贵。
  • 理论稀疏必须能转化为真实 GPU kernel 加速。

DSA 的原理：先找，再注意力

DSA 由两个组件组成：lightning indexer 负责给当前 query token 与历史 token 的相关性打分；fine-grained token selection 根据分数选择 top-k 历史 KV 条目。主注意力只对这些条目做 softmax 与加权求和。

公式

It,s = Σj=1..HI wIt,j · ReLU(qIt,j · kIs)

St = Top-k(It,:)
ut = Attn(ht, {cs | s ∈ St})

图中下三角代表因果可见历史，绿色格子代表被某个 query 选中的历史位置。DSA 的主注意力只在绿色位置上认真计算。

为什么叫 fine-grained

它不是只选一个固定窗口，也不是只能选连续 block。V3.2 的 DSA 可以在 token-level KV entry 上选择非连续历史，因此更适合代码定义、系统约束、工具输出、跨文档引用这类“远处但关键”的信息。

DeepSeek-V3.2 里的层结构

DeepSeek-V3.2 仍是 decoder-only Transformer + MoE。DSA 并不是单独一层替代整个模型，而是嵌入 MLA attention 内部。

训练策略

DSA 的训练分两步。第一步 dense warm-up：冻结主模型，只训练 indexer，让它用 KL loss 模仿 dense attention 的分布，1000 steps，总计约 2.1B tokens。第二步 sparse training：启用 top-k 选择，训练全模型适应稀疏模式，top-k 为 2048，15000 steps，总计约 943.7B tokens。

给一个 system prompt，它怎么自回归生成 token

System: 你是一个严谨的中文技术导师。回答要先给直觉，再给公式，最后给局限。
User: 用三句话解释 DeepSeek DSA。
Assistant:

system prompt 不会作为外部程序直接控制模型。它会被 chat template 包装成带角色标记的 token 序列，然后和用户问题一起进入模型。

ATokenize：system/user/assistant 标记和文本都变成 token id。
BPrefill：所有 prompt token 经过每层 Transformer，写入 KV 与 indexer cache。
CIndex：生成下一个 token 时，indexer 对历史位置打分。
DAttend：主注意力读取 top-k 历史条目，得到 hidden。
ESample：LM head 输出 logits，softmax 后采样或 argmax，追加 token，循环。

system prompt 的 token 会留在历史 cache 里。如果某条约束在后续仍然相关，indexer 可以把它选入 top-k，让主注意力重新读取。风险也在这里：DSA 依赖 indexer 的召回能力；训练中用 dense attention 分布监督 indexer，是为了降低“漏掉早期关键约束”的概率。

DeepSeek-V4 的 CSA/HCA

如果说 V3.2 的 DSA 是“历史 token 仍然逐个存在，但每次只挑 top-k 来读”，那么 V4 的 CSA/HCA 又往前走了一步：先把很多历史 token 合并成更少的压缩 KV 条目，再决定是稀疏读取还是全量读取。

关键区别很简单：MLA 主要压缩每个 token 存下来的表示宽度；CSA/HCA 主要压缩历史序列的长度。也就是说，不只是“每本书页更薄”，而是“很多页先做成一张摘要卡”。

Sliding Window

最近的 token 保留细节，直接看局部窗口。

V4 报告里额外 sliding-window branch 的窗口大小设为 128，用来补住压缩历史丢掉的近邻细节。

CSA

每 4 个历史 token 压成 1 个 KV 条目，再用 DSA 的 lightning indexer 选 top-k 压缩条目。

DeepSeek-V4-Pro：CSA 压缩率为 4，indexer heads 为 64，indexer head dim 为 128，attention top-k 为 1024。

HCA

每 128 个历史 token 压成 1 个 KV 条目，然后在这个很短的压缩序列上做 dense attention。

HCA 不再做 DSA top-k，因为历史已经被重度压短；它更像一条便宜的全局摘要通道。

为什么要同时用 CSA 和 HCA

CSA 和 HCA 是两种不同的折中。CSA 压缩温和，保留的压缩条目更多，所以还需要 DSA 再挑一批最相关的条目来读；同时，CSA 会把少量 sliding-window KV entries 与选中的压缩条目合并，补住局部细节。HCA 压缩激进，历史条目变得很少，所以可以 dense 地全读；它适合提供非常便宜的全局覆盖。

DeepSeek-V4 把 CSA 和 HCA 交错使用，而不是只选一种。V4-Pro 的公开设置是：61 层、hidden dim 7168，前两层使用 HCA，后续层 CSA/HCA interleaved；CSA 压缩率 4、top-k 1024，HCA 压缩率 128。V4-Flash 则是 43 层、hidden dim 4096，前两层 pure sliding window，后续层 CSA/HCA interleaved；CSA top-k 512，HCA 压缩率同样是 128。

V3.2 DSA：历史长度 L，主注意力读 top-k 个原始/latent KV 条目。
V4 CSA：历史长度先约变成 L/4，再从压缩条目中 DSA top-k。
V4 HCA：历史长度先约变成 L/128，再 dense 读取这些重压缩条目。

优点、代价与未来

DSA 带来的收益

  • 主注意力从 O(L²) 变成 O(Lk)。
  • 选择是 query-dependent，比固定窗口更灵活。
  • 能承接 DeepSeek 既有 MLA 路线，迁移风险低。
  • 围绕 FP8、MQA mode、top-k mask、KV cache 做了工程优化。

仍然存在的代价

  • lightning indexer 自身仍有 O(L²) 打分结构。
  • top-k 会造成不规则访存，需要专门 kernel。
  • 必须经过 warm-up 与 sparse training。
  • 稀疏选择有漏召回风险，尤其是很早但关键的约束。

DeepSeek-V4 的方向说明，DSA 不是终点，而是长上下文注意力中的检索核心。V4 的 CSA 先沿序列维度压缩 KV cache，再使用 DSA；HCA 更重度压缩后做 dense attention；再加 sliding window branch 保留最近 token 的未压缩细节。目标从 V3.2 的 128K 走向 V4 的 1M context。

未来趋势 = hidden 维度压缩 + 序列维度压缩 + learned sparse retrieval + 局部精读
